- 対象プロジェクト: GetScheduledShipmentQuantity（出荷予定数取得）
- 作成者: Claude（設計） → Antigravity（実装） → Gemini（レビュー）

- 種別: 追加機能（既存関数の削除・書き換えは行わない。追記のみ）

---

## 1. 背景・目的

2026-07-22朝、ネクストエンジンAPIのトークン不整合（エラーコード 002002）により
`warmUpNextEngineConnection()` および `updateShippingData()` が連続して失敗した。
GAS側からは自動復旧できず、ブラウザでの手動再認証（`generateAuthUrl()` → ログイン）
が必要となる障害だったが、発見が遅れた。

再発時に即座に気づけるよう、失敗発生時にメールで通知する機能を追加する。

---

## 2. 通知ポリシー（確定事項）

| 発生元 | 重要度 | 送信タイミング |
|---|---|---|
| `warmUpNextEngineConnection()` の失敗（catch節到達） | 警告（WARNING） | 発生の都度、毎回送信（抑制なし） |
| `updateShippingData()` 内、3回リトライ失敗確定時 | 緊急（CRITICAL） | 発生の都度、毎回送信（抑制なし） |

> 発生頻度が低い（2025年10月以降、今回まで発生なし）ため、
> 重複通知の抑制ロジックは実装しない。シンプルさを優先する。

**通知先メールアドレス**
- スクリプトプロパティは追加しない。
- `Session.getEffectiveUser().getEmail()` を使用し、
  トリガー実行者（＝スクリプト所有者）のメールアドレスを動的に取得する。

## 4. 新規ファイル：`16_通知.gs`

### 4.1 実装する関数

```
/**
 * エラー発生時にスクリプト所有者へメール通知を送信する共通関数
 *
 * 【目的】
 * ネクストエンジンAPI連携処理（ウォームアップ・本番データ取得）で
 * 復旧困難なエラーが発生した際、即座に開発者へメールで知らせる。
 *
 * 【動作】
 * - 通知先はSession.getEffectiveUser().getEmail()で動的取得する
 *   （スクリプトプロパティ・ハードコードは使用しない）
 * - severityに応じて件名の接頭辞を変える（[警告] / [緊急]）
 * - MailApp.sendEmail()を使用する
 * - メール送信自体が失敗した場合も、呼び出し元の処理を止めないよう
 *   try-catchで囲み、ログ出力のみに留める
 *
 * @param {string} severity - 'WARNING'（警告） または 'CRITICAL'（緊急）
 * @param {string} subject - 件名（重要度接頭辞を除いた本題部分）
 * @param {string} body - 本文（発生時刻・エラーメッセージ・関数名などを含める）
 * @returns {void}
*/
function notifyByEmail(severity, subject, body) {
    // 実装方針:
    // 1. severityが'CRITICAL'なら件名接頭辞'【緊急】'、
    //    'WARNING'なら'【警告】'とする（不明な値は'【通知】'とし、ログに警告を出す）
    // 2. Session.getEffectiveUser().getEmail() でメールアドレスを取得
    //    取得できない場合（空文字等）はconsole.errorでログ出力し、送信をスキップする
    //    （ここで例外を投げて呼び出し元の処理を止めてはならない）
    // 3. MailApp.sendEmail(recipient, fullSubject, fullBody) を実行
    // 4. try-catchで全体を囲み、送信失敗時もconsole.errorのみでリターンする
}
```

   残りクォータが極端に減っていないか確認する

## 7. 注意事項

- `MailApp.sendEmail()` の送信数上限は、Googleアカウントの種別により
  1日あたり100通（個人Gmail）〜1500通（Google Workspace）まで異なる。
  今回は「毎回送信・抑制なし」の方針だが、
  仮に短時間に大量のトリガー失敗が連鎖した場合、上限到達により
  通知そのものが送れなくなるリスクがある点は認識しておくこと
  （今回はポリシー変更不要、注意点としての共有のみ）。
- 既存関数（`warmUpNextEngineConnection`、本番側のリトライ処理）の
  ロジック・戻り値・エラーハンドリングの流れ自体は一切変更しないこと。
  追記のみでロールバック可能な状態を維持する。
- ファイル冒頭のヘッダーコメント（JSDoc）は、既存の説明文を削除せず、
  変更点を追記する形で更新すること。
